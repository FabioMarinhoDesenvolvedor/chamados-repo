## Entregue — Novo papel OPERATOR
- Enum `Role` agora `ADMIN | USER | OPERATOR` (shared + Prisma). Migration
  `20260630102219_add_operator_role` (`ALTER TYPE "Role" ADD VALUE 'OPERATOR'` — única, OPERATOR
  não é DEFAULT). Helper único `isStaffRole` (ADMIN∪OPERATOR) em `@chamados/shared`.
- **OPERATOR** (equipe de atendimento): vê todos os chamados, **assume só para si**, altera
  status, comenta e resolve (RESOLVED). **Não** cria chamados, **não** faz triagem, **não**
  conclui (CLOSED), **não** vê a avaliação (rating) e **não** acessa Usuários/Relatórios/
  Departamentos/Backup.
- Backend (`tickets`): `create` → `@Roles('USER','ADMIN')`; `status`/`assign` →
  `@Roles('ADMIN','OPERATOR')`; `assign(user)` valida assignee staff + OPERATOR só a si;
  `close` exige solicitante ou ADMIN; `ensureCanView` libera staff; `hideForUser`→`hideByRole`
  (OPERATOR esconde só rating).
- Frontend: `isStaff` (visão de atendimento) vs `isAdmin` (administrativo). Dashboard/Detalhe
  usam `isStaff`; OPERATOR vê botão **"Assumir para mim"** (não chama `GET /users`); "Novo
  chamado" oculto p/ OPERATOR (nav + dashboard + guard na rota). `ROLE_LABEL.OPERATOR='Operador'`
  → admin já pode criar operadores no `UsersPage` (select usa `ROLES`).
- Testes (node:test): `tickets.service.spec` (assign/close/hideByRole) + `roles.spec`
  (`isStaffRole`). **Total: 15 (api) + 1 (shared), todos passando.**
- Docs atualizadas: `business-rules.md`, `01-visao-geral.md`, `04-api-referencia.md`.
- **Deploy (você):** aplicar a migration — `npm run db:generate && npm run db:deploy -w @chamados/api`
